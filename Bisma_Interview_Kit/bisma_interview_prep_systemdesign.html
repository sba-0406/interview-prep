How to Answer a System Design Question

Interviewers don't expect a perfect system. They want to see how you think, ask clarifying questions, and make tradeoffs consciously. Follow this framework every time.

01The 6-Step SD Framework
Must KnowAlways First
✓
▼
The Framework — Follow This Every Time
1
Clarify requirements (2–3 min) — Ask before designing. "Is this read-heavy or write-heavy? How many users? What's the expected QPS? Do we need real-time or eventual consistency?" Shows senior thinking. Never jump straight to drawing boxes.
2
Estimate scale (1–2 min) — Back-of-envelope math. "1M users × 10 requests/day = 10M requests/day = ~116 QPS. 1M users × 1KB profile = 1GB storage." Shows you think about scale.
3
Define the API (2 min) — What endpoints/functions does the system expose? This forces you to understand what the system actually needs to do.
4
High-level design (5–7 min) — Draw the major components: clients, API servers, databases, caches, queues. Don't go deep yet. Get the skeleton right.
5
Deep dive (10–15 min) — The interviewer will ask you to go deep on specific parts. Discuss tradeoffs: SQL vs NoSQL, cache strategy, sharding approach, handling failures.
6
Identify bottlenecks & scale (3–5 min) — Where will your system break at 10× load? "The DB will be the bottleneck at scale — I'd add read replicas and cache frequently accessed data in Redis."
Clarifying Questions to Always Ask
→Scale — How many users? Requests per second? Data size? Read/write ratio?
→Consistency — Strong consistency (banking) or eventual consistency (social feed) acceptable?
→Availability — 99.9% or 99.99% uptime? (99.99% = 52 min downtime/year)
→Latency — What's the acceptable response time? Real-time (chat) vs near-real-time (analytics)?
→Geography — Single region or global? Affects CDN and database replication decisions.
What Interviewers Actually Evaluate
Problem clarification — do you ask smart questions or jump in?
Tradeoff awareness — do you know why you made each choice?
Breadth — can you name the right components (cache, queue, CDN)?
Depth — can you explain how one component works internally?
Communication — can you explain your thinking clearly while drawing?

Scalability — Vertical vs Horizontal

The foundational question in every system design: how do you handle 10× more traffic?

01Vertical vs Horizontal Scaling
Must Know
✓
▼
Vertical Scaling (Scale Up) Horizontal Scaling (Scale Out)
What Bigger machine — more CPU, RAM, SSD More machines — distribute load
Limit Hard ceiling — biggest server has limits Theoretically unlimited
Cost Exponentially expensive at top end Linearly expensive — add commodity servers
Downtime Requires restart to upgrade Add servers without downtime
Complexity Simple — same code, bigger box Complex — distributed system problems
Single point of failure Yes — one machine fails = down No — one server fails, others handle it
Use for Databases (initially), simple apps Stateless web servers, microservices
Stateless vs Stateful — The Key to Horizontal Scaling
→Stateless servers — don't store any user data locally. Any server can handle any request. Easy to horizontally scale — just add more.
→Stateful servers — remember client data (session info). User must always hit the same server (sticky sessions). Hard to scale — sessions must be shared via Redis/DB.
→Your Express + JWT apps are stateless — any server can verify the JWT. This is why JWT scales better than sessions.
→Databases are inherently stateful — harder to horizontally scale (need sharding, replication).
How to say it in an interview

"I'd start with vertical scaling because it's simple — no code changes needed. But vertical has a hard ceiling and a single point of failure. For the web tier, I'd move to horizontal scaling behind a load balancer since my Express app is stateless — any instance can handle any request. For the database, I'd start with read replicas to scale reads, then consider sharding if writes become the bottleneck."

Load Balancing

Distributes incoming traffic across multiple servers. Layer 4 vs Layer 7, algorithms, and health checks.

01Load Balancer — Types & Algorithms
Must Know
✓
▼
Load Balancer

A server that sits in front of your application servers and routes incoming requests to one of them based on an algorithm. Acts as traffic cop — prevents any single server from being overwhelmed.

Layer 4 vs Layer 7
Layer 4 (Transport) Layer 7 (Application)
Operates at TCP/UDP level HTTP/HTTPS level
Sees IP addresses and ports only Full HTTP request — URL, headers, cookies
Routing based on IP + port URL path, hostname, headers, cookies
Speed Faster — less inspection Slower — but smarter routing
Example AWS NLB, HAProxy TCP mode AWS ALB, Nginx, HAProxy HTTP mode
Use case Simple TCP routing, gaming Route /api to backend, /static to CDN
Load Balancing Algorithms
→Round Robin — requests distributed in sequence: server 1, 2, 3, 1, 2, 3... Simple. Doesn't account for server load.
→Weighted Round Robin — servers with more capacity get more requests. Server A (4 cores) gets 2× requests vs Server B (2 cores).
→Least Connections — next request goes to server with fewest active connections. Better for varying request durations.
→IP Hash — same client IP always goes to same server (sticky sessions without actual sessions). Useful for WebSocket connections.
→Least Response Time — combines least connections + fastest response time. Most sophisticated.
Health Checks
→Load balancer periodically pings each server (GET /health) to check if it's alive
→If a server fails health check → removed from rotation automatically
→When server recovers → added back to rotation
→Your /health endpoint should check DB connectivity, not just return 200
Likely Interview Questions

What is sticky session? When would you need it?
What happens if the load balancer itself goes down?

Database Scaling — Replication & Sharding

The database is almost always the bottleneck at scale. Know replication for reads and sharding for writes.

01Replication — Scale Reads
Must Know
✓
▼
Replication

Keep copies of the database on multiple servers. One primary (master) handles writes. One or more replicas (slaves) handle reads. All writes to primary are asynchronously copied to replicas.

// Master-Slave Replication pattern Client → Write (INSERT/UPDATE/DELETE) → PRIMARY (Master) ↓ async replication Client → Read (SELECT) → REPLICA 1, REPLICA 2, REPLICA 3 // Benefits: // - Read throughput scales linearly — add more replicas for more read capacity // - Primary is less loaded — only handles writes // - Replica can be used for backups without impacting primary // Problems: // - Replication lag — replica may be slightly behind primary (eventual consistency) // "Read your own writes" problem: user writes, immediately reads — may get stale data // - Primary failure — need to promote a replica to primary (failover) // - Writes still bottleneck on primary
02Sharding — Scale Writes
Must Know
✓
▼
Sharding (Horizontal Partitioning)

Split your data across multiple database servers (shards). Each shard holds a subset of the data. Queries only go to the relevant shard. Scales both reads and writes — each shard handles less load.

// Sharding example — users split across 4 shards by userId Shard 0: users where userId % 4 == 0 (user 0, 4, 8, 12...) Shard 1: users where userId % 4 == 1 (user 1, 5, 9, 13...) Shard 2: users where userId % 4 == 2 Shard 3: users where userId % 4 == 3 // Shard key choices: // userId — good for user data, even distribution // geoHash — good for location-based data (users in same region together) // date — bad! all current writes go to today's shard (hot spot) // Problems with sharding: // Cross-shard queries — JOIN across shards is expensive/impossible // Resharding — adding a new shard means moving data (painful) // Hotspots — popular shard gets overwhelmed (e.g., celeb's data on one shard)
Sharding vs Partitioning
→Horizontal partitioning (sharding) — split rows across multiple DBs. Different rows on different machines.
→Vertical partitioning — split columns across DBs. User's personal data in one DB, activity log in another.
How to say it in an interview

"I'd start with replication — add read replicas to scale reads, since most apps are 80-90% reads. If writes become the bottleneck, I'd add sharding. The shard key choice is critical — I'd use userId for user data to ensure even distribution. Cross-shard queries become impossible so I'd design schemas to avoid them — keeping all of a user's data on the same shard."

Likely Interview Questions

What is replication lag and how does it affect consistency?

Caching at Scale & CDN

Caching is the #1 performance tool in system design. Know the cache hierarchy, eviction policies, and CDN.

01Cache Hierarchy & Eviction Policies
Must Know
✓
▼
// Cache hierarchy (fastest to slowest): L1 Cache (CPU) — nanoseconds, ~1MB L2 Cache (CPU) — nanoseconds, ~10MB RAM (In-memory) — microseconds, ~GB ← your Moodify in-memory cache Redis (Distributed) — sub-millisec, ~TB ← shared cache across servers CDN (Edge cache) — milliseconds, ~TB ← static files cached globally Database — 1-100ms, ~PB ← source of truth
Cache Eviction Policies
→LRU (Least Recently Used) — evict the item not accessed for the longest time. Most common. Good for temporal locality.
→LFU (Least Frequently Used) — evict the item accessed fewest times overall. Good when popular items should stay forever.
→FIFO — evict the oldest item regardless of access. Simplest. Often not ideal.
→TTL — evict after a fixed time. Your Moodify cache uses this. Simple, prevents stale data.
Cache Problems
→Cache stampede (thundering herd) — popular cache entry expires. Thousands of requests hit DB simultaneously. Fix: use mutex/lock while regenerating, or add jitter to TTL.
→Cache penetration — requests for data that doesn't exist, bypassing cache every time. Fix: cache null results too (with short TTL).
→Cache avalanche — many cache entries expire at same time → DB overloaded. Fix: stagger TTL values (add random jitter).
02CDN — Content Delivery Network
Must Know
✓
▼
CDN

A network of geographically distributed edge servers that cache static content (images, JS, CSS, videos) close to users. Instead of serving from one origin server in Hyderabad to a user in the US (high latency), serve from an edge server in New York (low latency).

→Static content — HTML, CSS, JS, images, videos. Cached at edge for hours/days.
→Dynamic content — personalised API responses. Not cached (or cached very briefly).
→Cache-Control headers — tell CDN how long to cache: max-age=86400 (1 day)
→Cache invalidation — push update to CDN when content changes. Use versioned filenames (main.v2.js) to avoid stale cache.
→Examples — Cloudflare, AWS CloudFront, Akamai, Fastly

Message Queues & Pub/Sub

Decouple services, handle traffic spikes, enable async processing. Know Kafka vs RabbitMQ and when to use queues.

01Message Queues — Why and How
Must Know
✓
▼
Message Queue

A component where producers send messages and consumers pick them up asynchronously. Producers and consumers are decoupled — they don't need to be running at the same time, and producer doesn't wait for consumer to finish.

Why Use a Message Queue?
→Decouple services — email service doesn't need to be available when user registers. Just put "send welcome email" in queue.
→Handle traffic spikes — 10,000 users submit applications simultaneously. Queue absorbs the spike, workers process at their own pace.
→Retry failed tasks — if email service is down, message stays in queue until it recovers.
→Async processing — user gets instant response, heavy work happens in background (AI assessment scoring).
Kafka vs RabbitMQ
Kafka RabbitMQ
Type Distributed log / event streaming Traditional message broker
Message retention Keeps messages for days/weeks (configurable) Deletes after consumption
Consumers Multiple consumer groups — each gets all messages One consumer gets each message
Throughput Millions of messages/sec Thousands of messages/sec
Use case Event streaming, analytics, audit log, real-time pipelines Task queues, job processing, RPC
Ordering Guaranteed within a partition FIFO per queue
Complexity Higher — needs Zookeeper/KRaft Simpler to set up
Pub/Sub vs Queue
→Queue — one message, one consumer. Task is done once. Used for: job processing, email sending.
→Pub/Sub — one message, many subscribers all receive it. Used for: notifications, event broadcasting, cache invalidation across services.
How to say it in an interview

"In the Darwinbox assessment platform, after a candidate submits their test, we could use a message queue: the API immediately returns 'submission received' while a worker processes and scores the assessment asynchronously. This decouples the submission API from the scoring logic. If the AI scoring service goes down, submissions stay in the queue and are processed when it recovers — no data loss."

Microservices vs Monolith

One of the most discussed architectural decisions. Know the tradeoffs — not just the benefits of microservices.

01Monolith vs Microservices — Honest Comparison
Must KnowAlways Asked
✓
▼
Monolith Microservices
Structure Single deployable unit — all code together Multiple independent services, each deployed separately
Development Simple — one codebase, one deploy Complex — multiple repos, multiple deploys, distributed debugging
Scaling Scale entire app even if only one part is the bottleneck Scale only the bottleneck service independently
Tech stack Single language/framework Each service can use different language (polyglot)
Failure One bug can crash entire app Service failure is isolated
Network calls Function calls — fast, in-process HTTP/gRPC calls — latency, failure points
Team size Works well for small teams Works well for large teams (Conway's Law)
Data One shared database Each service owns its own database
When to Use Each
→Start with monolith — faster to build, easier to debug, no network overhead. Most startups and small teams should start here.
→Move to microservices when — team has grown too large for one codebase, specific parts need independent scaling, deploy cycles conflict.
→Common mistake — going microservices too early. Premature optimisation. "We're not Google yet."
Microservices Challenges
→Distributed tracing — one request spans multiple services. Need tools like Jaeger, Zipkin to trace.
→Service discovery — how does Service A find Service B? Use service registry (Consul, Kubernetes DNS).
→Data consistency — each service has its own DB. No cross-service transactions. Use Saga pattern.
→Network failures — every inter-service call can fail. Need retry logic, circuit breakers (like your ResilientAIService).
How to say it in an interview

"Darwinbox as an enterprise SaaS would likely be microservices — separate services for recruitment, payroll, attendance, each scalable independently. My internship project was a monolith within the recruitment module — one Node.js app with 8 controllers and 9 service modules. That's the right approach for a bounded feature set. I'd move to microservices when different modules need independent deployment cycles or scaling — for example, the AI assessment scoring service is CPU-intensive and should scale independently from the job-posting service."

Must Know

API Gateway

A reverse proxy that sits between clients and your microservices. All external requests go through it. It handles cross-cutting concerns so each service doesn't have to implement them individually.

// Without API Gateway — each service handles everything Client → Auth Service (has its own auth, rate limiting, logging) Client → Job Service (has its own auth, rate limiting, logging) Client → Payment Service (has its own auth, rate limiting, logging) // Duplicated logic everywhere! // With API Gateway Client → API Gateway → Auth Service → Job Service → Payment Service // API Gateway handles: // ✓ Authentication — verify JWT once, pass user context to services // ✓ Rate limiting — per client, per endpoint // ✓ SSL termination — HTTPS at gateway, HTTP internally // ✓ Request routing — /api/jobs → job-service, /api/payments → payment-service // ✓ Load balancing — across service instances // ✓ Request/Response transformation — protocol translation // ✓ Logging & monitoring — single place for all request logs // ✓ Caching — cache common responses // Examples: AWS API Gateway, Kong, Nginx, Traefik, Express Gateway

Consistent Hashing

Solves the resharding problem. Used by Redis Cluster, Cassandra, DynamoDB, and CDNs for distributing data.

01Consistent Hashing — The Problem It Solves
Must Know
✓
▼
The Problem with Simple Hashing
// Simple hash: serverIndex = hash(key) % numServers // 3 servers: key "user123" → hash = 300 → 300 % 3 = 0 → Server 0 // Problem: add a 4th server // key "user123" → hash = 300 → 300 % 4 = 0 (same!) // key "user456" → hash = 456 → 456 % 3 = 0, but 456 % 4 = 0 (moved!) // 75% of all keys move to different servers when you add/remove ONE server! // Massive cache misses, data needs to move
Consistent Hashing — The Solution
→Imagine a ring (circle) of hash values from 0 to 2^32-1
→Servers are placed at positions on the ring by hashing their IP/name
→A key is assigned to the first server clockwise from its hash position on the ring
→Adding a server — only keys between the new server and its predecessor move. ~1/N keys affected.
→Removing a server — only keys on that server move to its successor. ~1/N keys affected.
→Virtual nodes — each physical server has multiple positions on the ring. Ensures even data distribution.
How to say it in an interview

"Consistent hashing solves the resharding problem. With naive modulo hashing, adding one server causes most keys to move — causing cache misses everywhere. With consistent hashing, servers and keys are mapped to a ring. Adding a server only moves ~1/N of all keys to the new server — minimal disruption. Redis Cluster and Cassandra use consistent hashing for distributing data across nodes."

// Short code generation — 7 characters, base62 (a-z A-Z 0-9) // 62^7 = 3.5 trillion combinations — enough for billions of URLs // Option A: Hash the long URL shortCode = base62(md5(longUrl)).substring(0, 7) // Problem: collision possible, same URL gives same code // Option B: Auto-increment ID + base62 encode (better) id = db.autoIncrement() // 1, 2, 3... shortCode = base62Encode(id) // "1" → "b", "1000000" → "4c92" // Database schema URLs table: id BIGINT PRIMARY KEY AUTO_INCREMENT short_code VARCHAR(7) UNIQUE INDEX long_url TEXT created_at TIMESTAMP expires_at TIMESTAMP user_id BIGINT // High-level architecture Client → Load Balancer → App Servers → Redis Cache → Database ↑ miss PostgreSQL (primary + replicas)
Step 5 — Deep Dive: Redirect Performance
// Redirect flow (must be <100ms): GET /abc123 1. Check Redis: GET "url:abc123" → hit → return 301 redirect (~1ms) 2. Miss → query DB → cache in Redis with TTL → return redirect (~10ms) // Cache most popular URLs in Redis (80/20 rule — 20% URLs = 80% traffic) // Redis can handle 115,000 reads/sec easily
Step 6 — Bottlenecks & Scale
Write bottleneck: Single DB for auto-increment is fine for 1,157/sec. At scale: use multiple ID generators (Twitter Snowflake — distributed unique IDs).
Read bottleneck: Redis cache absorbs 99% of reads. DB only sees cache misses.
Geography: Add CDN and edge caches for redirect — serve from nearest region.

Design a Rate Limiter

Tests distributed systems thinking. Token bucket is the answer — you already implemented throttling in Darwinbox.

01Distributed Rate Limiter Design
Often Asked
✓
▼
// Problem with in-memory rate limiter in multi-server setup: Server 1: User A made 3 requests → allows 7 more Server 2: User A made 3 requests → allows 7 more (doesn't know about Server 1!) // User effectively gets 20 requests instead of 10! // Solution: Redis as shared counter (all servers share one Redis) // Token Bucket with Redis: async function isAllowed(userId, limit=100, windowSec=60) { const key = `ratelimit:${userId}`; // MULTI/EXEC for atomic operation const pipeline = redis.pipeline(); pipeline.incr(key); pipeline.expire(key, windowSec); const [count] = await pipeline.exec(); if (count > limit) { // Set Retry-After header const ttl = await redis.ttl(key); throw new RateLimitError(ttl); } return true; } // Response headers for good API design: // X-RateLimit-Limit: 100 // X-RateLimit-Remaining: 45 // X-RateLimit-Reset: 1714086400 (Unix timestamp when window resets) // Retry-After: 30 (seconds until next request allowed, on 429)
→Rate limit by — IP address (unauthenticated), user ID (authenticated), API key (B2B)
→Different limits per endpoint — login: 5/min, general API: 100/min, AI endpoints: 10/min
→Sliding window log — store timestamps of each request in Redis sorted set. More accurate than fixed window but uses more memory.
How to connect to your experience

"I implemented request throttling in my ResilientAIService using the token bucket pattern in memory. For a distributed system with multiple servers, you'd move that shared counter to Redis so all server instances see the same count. The logic is identical — just replace the in-process counter with a Redis INCR + EXPIRE."

Time & Space Complexity

You've solved 600+ problems — you know this practically. Now express it clearly in an interview.

01Big O — Common Complexities
Must KnowDSA
✓
▼
// COMMON COMPLEXITIES — fastest to slowest: O(1) Constant — HashMap get, array index access O(log n) Logarithmic — Binary search, BST operations, heap push/pop O(n) Linear — Linear scan, sum of array O(n log n) Linearithmic — Merge sort, heap sort, most efficient sorts O(n²) Quadratic — Bubble sort, nested loops over same array O(2^n) Exponential — Recursive fibonacci (unoptimised), subset generation O(n!) Factorial — Permutations of n elements // QUICK PATTERN RECOGNITION: // One loop over n elements → O(n) // Loop inside a loop over n → O(n²) // Halving the problem each step → O(log n) // One loop + binary search inside → O(n log n) // Recursive with two branches → often O(2^n), use memoization → O(n) // SPACE COMPLEXITY: // Recursive calls use stack space — depth of recursion = space // Storing n elements = O(n) space // In-place algorithm = O(1) extra space (modify input)
How to Analyse in an Interview
→Count loops — each nested loop multiplies complexity
→Ignore constants — O(2n) = O(n), O(n + n²) = O(n²)
→Dominant term wins — always take the largest growing term
→Always state BOTH time and space complexity when done solving
How to say it in an interview

"My solution uses one pass over the array so it's O(n) time. I use a HashMap to store seen elements, which is O(n) space in the worst case. If we wanted O(1) space, we could sort the array first (O(n log n) time) and use two pointers."

Arrays, LinkedLists & HashMaps

The three most-used data structures. Know their internals, tradeoffs, and when to choose each.

01When to Use Which — With Complexity
Must KnowDSA
✓
▼
Operation Array LinkedList HashMap
Access by index O(1) ✓ O(n) ✗ N/A
Search (unsorted) O(n) O(n) O(1) ✓
Insert at end O(1) amortized O(1) with tail pointer O(1) amortized
Insert at middle O(n) — shift O(1) — relink N/A

→Use Array when — you need index-based access, or cache performance matters (contiguous memory). Default choice for most problems.

→Use HashMap when — fast lookup by key, counting frequencies, checking membership, caching results. O(1) average for get/put/delete.
HashMap Collision Handling — Two Methods
→Chaining — each bucket is a linked list. Collisions appended to the list. Java HashMap uses this (then converts to tree at 8 entries).
→Open Addressing — if bucket occupied, probe to next slot. Less memory overhead but more complex. Python dict uses this.

BST — Binary Search Tree Properties
→Left child < parent < right child — always
→Search, insert, delete: O(log n) for balanced tree, O(n) for skewed (degenerate) tree
→Inorder traversal of BST gives sorted sequence — useful for finding kth smallest
Balanced vs Unbalanced Trees
→Balanced BST — height is O(log n). Search is O(log n). Examples: AVL tree, Red-Black tree.
→Unbalanced BST — inserting sorted values (1,2,3,4,5) creates a linked list. Height O(n). Search degrades to O(n).
→Java's TreeMap and TreeSet use Red-Black tree — self-balancing, always O(log n).
Other Important Trees
→Heap — complete binary tree. Min-heap: parent ≤ children. Max-heap: parent ≥ children. O(log n) insert/delete, O(1) peek. Used for priority queue.
→Trie — prefix tree. Each node = one character. Used for autocomplete, spell check. O(L) operations where L = word length.
→Segment Tree — range queries (range sum, range min) in O(log n).

BFS (Breadth-First Search) DFS (Depth-First Search)
Data structure Queue (FIFO) Stack (or recursion)
Order Level by level — nearest nodes first Go as deep as possible before backtracking
Space O(w) — w = max width of graph O(h) — h = max depth/height
Shortest path? Yes — in unweighted graph, BFS guarantees shortest path No — doesn't guarantee shortest path
Use for Shortest path, level-order, nearby nodes first Path existence, cycle detection, topological sort, maze solving
Time complexity O(V + E) O(V + E)
Common Graph Problems — Know What Algorithm to Use
→Shortest path (unweighted) — BFS
→Shortest path (weighted) — Dijkstra's algorithm (uses min-heap)
→Detect cycle — DFS with visited + recursion stack tracking
→Topological sort — DFS or Kahn's algorithm (BFS-based). Used for build order, course prerequisites.
→Connected components — DFS or BFS from each unvisited node
→Bipartite check — BFS with 2-coloring
Graph Representations
→Adjacency List — HashMap<Node, List<Node>>. Space O(V+E). Good for sparse graphs. Most common.
→Adjacency Matrix — boolean[][]). Space O(V²). O(1) edge lookup. Good for dense graphs.

Key Concepts
→Stable sort — equal elements maintain their original relative order. Merge Sort is stable; Quick Sort is not.
→QuickSort worst case O(n²) — happens when pivot is always smallest/largest (sorted input). Fixed by randomised pivot or median-of-three.
→TimSort — hybrid of Merge Sort + Insertion Sort. Java Arrays.sort() and Collections.sort() use TimSort. O(n) on nearly-sorted data.
→When to use Insertion Sort — small arrays (<10 elements) or nearly-sorted. TimSort uses Insertion Sort for small runs internally.
→Counting Sort / Radix Sort — O(n+k) where k is range. Non-comparison based. Use when values are in a small, known range.

Dynamic Programming

Break a problem into overlapping subproblems, solve each once, store the result (memoization/tabulation). Two conditions: Optimal Substructure (optimal solution built from optimal sub-solutions) + Overlapping Subproblems (same sub-problems solved repeatedly).

How to Recognise a DP Problem
→Question asks for "minimum/maximum/count/ways" — strong DP signal
→Brute force is exponential (recursive tree with repeated subproblems)
→Can the problem be broken into smaller similar problems?
The Approach — 4 Steps
// Step 1: Define the state // dp[i] = minimum cost to reach step i // dp[i][j] = minimum edit distance between str1[0..i] and str2[0..j] // Step 2: Recurrence relation (transition) // Fibonacci: dp[n] = dp[n-1] + dp[n-2] // Climbing stairs: dp[i] = dp[i-1] + dp[i-2] (can take 1 or 2 steps) // Knapsack: dp[i][w] = max(dp[i-1][w], dp[i-1][w-weight[i]] + value[i]) // Step 3: Base cases // dp[0] = 0, dp[1] = 1 (fibonacci) // dp[0][j] = j, dp[i][0] = i (edit distance) // Step 4: Top-down (memoization) vs Bottom-up (tabulation) // Top-down: recursion + HashMap cache (natural, easier to think) // Bottom-up: iterate and fill dp array (no stack overflow, usually faster) // Example: Fibonacci // Naive recursion: O(2^n) // With memoization: O(n) Map memo = new HashMap<>(); long fib(int n) { if (n <= 1) return n; if (memo.containsKey(n)) return memo.get(n); long result = fib(n-1) + fib(n-2); memo.put(n, result); return result; }
Common DP Problem Patterns
→1D DP — Fibonacci, climbing stairs, house robber, coin change
→2D DP — Longest Common Subsequence, Edit Distance, 0/1 Knapsack
→String DP — palindrome partition, regex matching
→Grid DP — unique paths, minimum path sum

Stack, Queue & Heap

Three data structures with very specific use cases. Know when each is the right tool.

01Stack, Queue & Priority Queue — Use Cases
Must KnowDSA
✓
▼
Stack — LIFO (Last In First Out)
→Operations: push (add top), pop (remove top), peek (view top) — all O(1)
→Use for: function call stack, undo/redo, bracket matching, DFS iterative, monotonic stack problems
→Classic problems: valid parentheses, next greater element, daily temperatures, largest rectangle in histogram
Queue — FIFO (First In First Out)
→Operations: enqueue (add back), dequeue (remove front), peek — all O(1)
→Use for: BFS traversal, task scheduling, message queues, sliding window problems
→Deque (ArrayDeque) — double-ended, add/remove from both ends. Use for sliding window maximum.
Priority Queue (Heap)
→Always gives min (or max) element in O(1). Insert/delete O(log n).
→Use for: Dijkstra's shortest path, K-th largest/smallest element, merge K sorted lists, task scheduling by priority
→Java: PriorityQueue (min-heap by default). For max-heap: new PriorityQueue<>(Collections.reverseOrder())
// When to use which — decision tree: // Need LIFO? → Stack // Need FIFO? → Queue // Need min/max efficiently? → PriorityQueue (Heap) // Need sorted order always? → TreeMap / TreeSet (Red-Black Tree) // Need fast lookup by key? → HashMap // Need sorted + fast lookup? → TreeMap // Classic interview pattern — Top K elements: // Find K largest elements in array — use min-heap of size K // O(n log K) time instead of O(n log n) for full sort
