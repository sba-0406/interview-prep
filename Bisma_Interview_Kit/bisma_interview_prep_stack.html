Event Loop Deep Dive

The #1 most asked Node.js question. Every Node.js company will ask this. Know every phase cold.

⭐ Both Darwinbox & Moodify run on this
01What is the Event Loop?
Must KnowAlways Asked
✓
▼
Event Loop

Node.js runs JavaScript on a single thread. The event loop is the mechanism that lets Node.js handle concurrent operations without blocking — it offloads I/O to the OS or libuv thread pool, then picks up callbacks when work is done.

6 Phases — Run In Order, Repeatedly
1. TIMERS — setTimeout() and setInterval() callbacks 2. PENDING I/O — I/O callbacks deferred from previous iteration 3. IDLE/PREPARE — internal only 4. POLL — retrieve new I/O events; execute I/O callbacks (your file reads, DB queries, HTTP requests land here) 5. CHECK — setImmediate() callbacks 6. CLOSE — socket.on('close') callbacks Between EVERY phase: drain microtask queues → process.nextTick() — highest priority, before Promises → Promise .then() callbacks — after nextTick queue is empty
Classic Output Order Question
console.log('1'); setTimeout(() => console.log('2 - setTimeout'), 0); setImmediate(() => console.log('3 - setImmediate')); Promise.resolve().then(() => console.log('4 - Promise')); process.nextTick(() => console.log('5 - nextTick')); console.log('6'); // Output: // 1 (sync) // 6 (sync) // 5 (nextTick — highest priority microtask) // 4 (Promise microtask) // 2 (timers phase — even delay=0 waits for event loop) // 3 (check phase — setImmediate)
How to say it in an interview

"Node.js uses the event loop to handle concurrency on a single thread. When you call fs.readFile(), libuv handles the actual I/O on a thread pool — the JS thread is free to serve other requests. When I/O completes, the callback enters the event loop queue and runs during the poll phase. Microtasks like nextTick and Promises have higher priority than regular callbacks and drain between phases. This is why Node.js is excellent for I/O-bound workloads — the thread is never blocked waiting for disk or network."

01Middleware — Complete Guide
Must KnowAlways Asked

Middleware

A function with access to req, res, and next(). It can execute code, modify req/res, end the request, or call next() to pass to the next middleware. This is the Chain of Responsibility design pattern.

// Standard middleware function logger(req, res, next) { console.log(req.method, req.url); next(); // pass to next — if you don't call this, request hangs! } // Error-handling middleware — 4 params, must be registered LAST function errorHandler(err, req, res, next) { res.status(err.statusCode || 500).json({ error: err.message }); } // Your Darwinbox request flow (order matters): app.use(cors()); // 1. CORS headers app.use(helmet()); // 2. Security headers app.use(express.json()); // 3. Parse JSON body app.use(morgan('dev')); // 4. Request logging app.use('/api', authMiddleware); // 5. JWT verification app.use('/api', rbacMiddleware); // 6. Role check app.use('/api', router); // 7. Route handlers app.use(errorHandler); // 8. Error handler — always last
In Your Darwinbox Project

Your JWT auth middleware verifies the token and attaches req.user = {userId, role}. Your RBAC middleware then checks if req.user.role is in the allowed roles for that route. If JWT is invalid → next(new Error('401')), if role is wrong → next(new Error('403')). The error handler at the end catches both and returns the right status code. Classic Chain of Responsibility.

Clustering & Worker Threads

How to use all CPU cores. Important for system design interviews.

01Cluster Module vs Worker Threads
Often Asked
✓
▼
Cluster Module Worker Threads
How Forks multiple OS processes Multiple threads in same process
Memory Separate memory per process Shared memory possible
Best for Scaling HTTP server across CPU cores CPU-intensive tasks (image processing, ML)
Crash isolation Yes — worker crash doesn't kill master No — thread crash can kill process
// Cluster — one process per CPU core const cluster = require('cluster'); if (cluster.isMaster) { require('os').cpus().forEach(() => cluster.fork()); cluster.on('exit', () => cluster.fork()); // restart dead workers } else { app.listen(3000); // each worker listens on same port } // Worker Thread — offload CPU work without blocking event loop const { Worker } = require('worker_threads'); const worker = new Worker('./heavy-compute.js'); worker.postMessage({ data: largeArray }); worker.on('message', result => res.json(result));
How to say it in an interview

"By default Node.js uses only one CPU core. The cluster module forks multiple processes — one per core — all sharing the same port. The OS distributes connections between them. In production I'd use PM2 which handles clustering automatically. Worker threads are for CPU-heavy tasks like running ML inference or parsing large CSVs — you offload the heavy work to a thread so the event loop stays free."

Aggregation Pipeline

You built MongoDB aggregation-powered leaderboards and analytics dashboards. This is your strongest MongoDB talking point.

⭐ You built leaderboards with this — explain it
01All Key Stages + Your Leaderboard Example
Must KnowAlways Asked
✓
▼
Aggregation Pipeline

A sequence of stages where each stage's output becomes the next stage's input. Like a Unix pipe. More powerful than SQL GROUP BY.

// KEY STAGES { $match: { status: 'shortlisted', score: { $gte: 60 } } } // filter early! { $group: { _id: '$jobId', avg: { $avg: '$score' }, total: { $sum: 1 } } } { $sort: { avg: -1 } } { $limit: 10 }, { $skip: 0 } { $project: { name: 1, score: 1, _id: 0 } } { $lookup: { from: 'jobs', localField: 'jobId', foreignField: '_id', as: 'job' } } { $unwind: '$job' } // flatten array from $lookup { $addFields: { rank: { $multiply: ['$score', 1.1] } } } { $count: 'totalResults' } // YOUR DARWINBOX LEADERBOARD (what it likely looks like): const leaderboard = await Application.aggregate([ { $match: { jobId: mongoose.Types.ObjectId(jobId) } }, // filter by job FIRST { $group: { _id: '$candidateId', totalScore: { $sum: '$score' }, attempts: { $sum: 1 } }}, { $lookup: { from: 'candidates', localField: '_id', foreignField: '_id', as: 'candidate' }}, { $unwind: '$candidate' }, { $project: { name: '$candidate.name', email: '$candidate.email', totalScore: 1, attempts: 1 }}, { $sort: { totalScore: -1 } }, { $limit: 10 } ]);
How to say it in an interview

"I built the candidate leaderboard using MongoDB's aggregation pipeline. The key performance insight: I put $match first to filter by jobId before grouping — this reduces the documents the $group stage has to process from the entire collection down to just that job's applications. Then $lookup joins candidate details, $project reshapes the output, and $sort + $limit gives the ranked top 10. Moving $match later would mean grouping all documents in the collection — much slower."

Transactions, ACID & MVCC

PostgreSQL has the strongest ACID implementation. MVCC is how it achieves high concurrency without locking reads.

01MVCC — How PostgreSQL Handles Concurrency
Must Know
✓
▼
MVCC — Multi-Version Concurrency Control

PostgreSQL never overwrites existing rows. When you UPDATE, it creates a new version of the row and marks the old one as expired. Readers see a consistent snapshot as of when their transaction started. Result: readers never block writers, writers never block readers.

-- Basic transaction BEGIN; INSERT INTO moods (user_id, mood) VALUES (1, 'happy'); UPDATE user_stats SET mood_count = mood_count + 1 WHERE user_id = 1; COMMIT; -- or ROLLBACK on failure -- Savepoints — partial rollback BEGIN; INSERT INTO songs VALUES (...); SAVEPOINT sp1; INSERT INTO recommendations VALUES (...); -- this might fail ROLLBACK TO sp1; -- keeps the song insert COMMIT; -- Isolation levels (default in PG is Read Committed) SET TRANSACTION ISOLATION LEVEL REPEATABLE READ; -- VACUUM — PostgreSQL periodically cleans old row versions -- AUTOVACUUM runs automatically in background
How to say it in an interview

"PostgreSQL uses MVCC — when you update a row, it keeps the old version for readers who started before the update. Each transaction sees a consistent snapshot. This means readers never block writers and vice versa — multiple users can read mood recommendations in Moodify while a background job updates recommendation scores, all without locking. PostgreSQL periodically runs VACUUM to clean up old row versions."

For immediate revocation (logout must work instantly), I'd maintain a Redis blacklist of invalidated refresh tokens."

// Your Moodify cache (likely similar to this) class SentimentCache { constructor(ttlMs = 30 * 60 * 1000, maxSize = 1000) { this.cache = new Map(); this.ttlMs = ttlMs; // 30 min TTL this.maxSize = maxSize; } get(mood) { const entry = this.cache.get(mood); if (!entry) return null; if (Date.now() - entry.timestamp > this.ttlMs) { this.cache.delete(mood); return null; // expired } return entry.value; } set(mood, recommendations) { if (this.cache.size >= this.maxSize) { // LRU-style: evict oldest entry this.cache.delete(this.cache.keys().next().value); } this.cache.set(mood, { value: recommendations, timestamp: Date.now() }); } } const cache = new SentimentCache(); // In route handler async function getMoodRecommendations(req, res) { const mood = req.query.mood; const cached = cache.get(mood); if (cached) return res.json({ source: 'cache', data: cached }); // Cache miss — call Gemini AI (~2 seconds) const recommendations = await geminiAI.recommend(mood); cache.set(mood, recommendations); res.json({ source: 'api', data: recommendations }); }
Limitations of In-Memory Cache
→Not shared — each Node.js process has its own cache. Multiple server instances = cache misses across processes
→Lost on restart — process restart clears all cached data
→Memory bound — limited by available RAM
→For multi-server production → use Redis (shared across all processes)

"I cached Gemini AI sentiment results in Moodify using an in-memory Map. Generator-Critic Architecture

A two-model LLM pipeline where a large, powerful model generates output, and a smaller, faster model evaluates quality. Only output that passes the critic reaches the user. Catches LLM hallucinations and low-quality outputs before they cause problems.

// Your Darwinbox implementation (conceptual) async function generateAssessmentQuestions(jobDescription, skillWeights) { const required = calculateRequired(skillWeights); const toGenerate = required * 2; // 2X BUFFER — generate double, keep best // STEP 1: GENERATOR — LLaMA 3.3-70B (large, creative, powerful) const rawQuestions = await groqAPI.chat({ model: 'llama-3.3-70b-versatile', messages: [{ role: 'user', content: buildGeneratorPrompt(jobDescription, toGenerate) }] }); const parsed = parseQuestionsJSON(rawQuestions); // STEP 2: CRITIC — LLaMA 3.1-8B (smaller, faster, cheaper) const approved = []; for (const q of parsed) { const eval = await groqAPI.chat({ model: 'llama-3.1-8b-instant', messages: [{ role: 'user', content: buildCriticPrompt(q, jobDescription) }] }); const { passes, score } = parseEval(eval); if (passes && score >= 0.7) approved.push({ ...q, score }); } // STEP 3: Filter — top N by quality score + per-skill quota return approved .sort((a, b) => b.score - a.score) .slice(0, required); }
Why This Architecture Works
→Quality control — LLMs hallucinate. Generator might produce ambiguous or irrelevant questions. Critic catches them before candidates see them.
→2X buffer — generate double to ensure enough pass filtering
→Cost + speed — critic uses 8B model (10x cheaper and faster than 70B)
→Separation of concerns — generator optimises for creativity; critic optimises for accuracy
Your 2-minute interview explanation

"I built a generator-critic LLM pipeline for dynamic MCQ generation in Darwinbox. LLaMA 3.3-70B generates questions from the job description — it's creative and context-aware. But LLMs can hallucinate: produce questions that sound plausible but are ambiguous or irrelevant. So LLaMA 3.1-8B acts as a critic — it evaluates each question for clarity, job relevance, and difficulty. I use a 2X buffer strategy: generate twice what's needed, critic filters them, and only questions that pass the quality threshold go into the assessment. This means candidates always get well-formed questions, and HR teams don't need to review every generated question manually. The critic uses the smaller 8B model because evaluation is simpler than generation — it's faster and about 10x cheaper."

RAG — Retrieval-Augmented Generation

LLMs have a knowledge cutoff and don't know your private data. RAG solves this: retrieve relevant documents from your knowledge base, inject them into the prompt as context, let the LLM answer using that context.

// RAG Pipeline // 1. Indexing (one-time) const embedding = await openai.embeddings.create({ input: document }); // [0.12, -0.45, ...] await vectorDB.upsert({ id: docId, vector: embedding, metadata: document }); // 2. Query (per request) const queryEmbedding = await openai.embeddings.create({ input: userQuestion }); const relevant = await vectorDB.query({ vector: queryEmbedding, topK: 5 }); const context = relevant.map(r => r.metadata).join(' '); // 3. Generate with context const answer = await llm.chat({ messages: [ { role: 'system', content: 'Answer based only on the provided context.' }, { role: 'user', content: `Context: ${context} Question: ${userQuestion}` } ] });
LLM Limitations — Know These
→Hallucination — confidently generates false information. Your critic model catches this for MCQs.
→Context window limit — LLaMA 3.3-70B has 128K token context. Long documents must be chunked for RAG.
→Knowledge cutoff — training data has end date. Solved by RAG for private/recent data.
→Non-determinism — same prompt → different outputs. Temperature=0 reduces but doesn't eliminate.
→Prompt injection — malicious user input hijacks model behaviour. Sanitise inputs before injecting into prompts.

// MOODIFY ARCHITECTURE Browser (HTML/CSS/JS frontend) ↓ HTTPS Render (Web Service) ├── Docker container: Node.js + Express app ├── Auto-deploys from GitHub on every push (CI/CD) ├── Provides: HTTPS, custom domain, env vars, logs └── Free tier: spins down after 15 min inactivity (~30s cold start) ↓ Pooled connection (PgBouncer) ↓ HTTPS API call Neon PostgreSQL Gemini AI (Google Cloud) ├── Serverless PostgreSQL └── Mood → song recommendations ├── Scales to ZERO when idle ├── Built-in PgBouncer pooling ├── Branching: dev/staging/prod branches └── Connection: postgresql://...neon.tech/moodify?sslmode=require // WHY THESE CHOICES: // Render: free tier, GitHub auto-deploy, Docker support, simple dashboard // Neon: free tier, scales to zero (no DB cost when idle), PgBouncer built-in // serverless = perfect for hobby projects with unpredictable traffic // Separation: app and DB in different services = can scale independently // KEY CONCEPTS: // Stateless app — Render can spin up/down containers freely // because state (data) lives in Neon, not in the container // Cold start — Render free tier spins down after 15 min // First request after idle takes ~30s while container spins up // Not a problem for portfolio project; paid tier has always-on
How to say it in an interview

"Moodify is deployed on Render as a containerised Node.js service. I separated app and database — app on Render, PostgreSQL on Neon — because they scale differently and should fail independently. Neon's serverless PostgreSQL scales to zero when idle so there's no compute cost when nobody's using the app. It has built-in PgBouncer connection pooling, which is critical — without it, each Render request would open a new DB connection and hit PostgreSQL's connection limit quickly. The app is stateless: Render can restart the container at any time without data loss because all state is in Neon."
