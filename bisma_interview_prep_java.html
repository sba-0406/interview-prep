JVM, JDK & JRE

First Java question in almost every interview. Know the difference deeply — including how JVM executes code internally.

01JVM vs JRE vs JDK — The Real Difference
Must Know
✓
▼
Term Full Form Contains Who uses it
JVM Java Virtual Machine Executes bytecode. Platform-specific (separate JVM for Windows, Linux, Mac). The engine that makes Java portable. Everyone running Java
JRE Java Runtime Environment JVM + core class libraries (java.lang, java.util, java.io etc). Everything needed to RUN Java programs. End users running Java apps
JDK Java Development Kit JRE + javac compiler + javadoc + jdb debugger + jar tool + profiling tools. Everything to WRITE and BUILD Java. Developers
The Complete Java Code Execution Flow

You write HelloWorld.java → javac (compiler) compiles it to HelloWorld.class (bytecode) → JVM loads the .class file → Class Loader loads into memory → Bytecode Verifier checks for security violations → Interpreter runs it line by line, OR JIT Compiler compiles hot code to native machine code → CPU executes.

How to say it in an interview

"JDK is for development — it includes the compiler (javac) along with JRE. JRE is for running programs — it includes the JVM and standard libraries. JVM is the actual engine — it's platform-specific, which is what allows Java to be 'write once, run anywhere'. The bytecode is platform-independent, but the JVM that runs it is platform-specific."

02JVM Architecture — What's Inside
Important
✓
▼
Class Loader Subsystem
→Bootstrap ClassLoader — loads core Java classes (java.lang.*, rt.jar). Written in native code, parent of all loaders.
→Extension ClassLoader — loads classes from jre/lib/ext.
→Application ClassLoader — loads your .class files from classpath. This is what loads your code.
→Delegation Model — each loader first delegates to its parent before trying to load itself. Prevents malicious classes overriding java.lang.String.
Runtime Data Areas (Memory Areas)
→Method Area (shared) — stores class metadata, static variables, method bytecode, constant pool. In Java 8+, this is called Metaspace (off-heap).
→Heap (shared) — all object instances and arrays. Divided into Young Gen and Old Gen. Garbage collected.
→JVM Stack (per thread) — each thread has its own stack. Each method call creates a Stack Frame containing local variables, operand stack, and frame data. Stack Frame is destroyed when method returns.
→PC Register (per thread) — holds address of current executing instruction.
→Native Method Stack (per thread) — for native (C/C++) method calls.
Execution Engine
→Interpreter — reads and executes bytecode line by line. Simple but slow — same code executed multiple times gets interpreted each time.
→JIT Compiler (Just-In-Time) — monitors which methods are "hot" (called frequently). Compiles those to native machine code. After compilation, native code runs directly — much faster. This is why Java performance improves over time ("warm-up").
→Garbage Collector — automatically frees unreachable objects from heap.

03Platform Independence — How It Actually Works
Important
✓
▼
→Java code compiles to bytecode — a middle-level, platform-neutral instruction set. Not machine code, not source code.
→The same .class file runs on any machine that has a JVM installed — Windows, Linux, Mac.
→The JVM itself IS platform-specific — there are different JVMs for different operating systems. The JVM handles translating bytecode to each OS's native code.
→"Write Once, Run Anywhere" (WORA) — Java's promise. One .jar file runs everywhere as long as the right JVM version is installed.
→Caveat — JNI (Java Native Interface) lets Java call native OS code, which breaks portability.
How to say it in an interview

"Platform independence comes from bytecode. When you compile Java, you get .class files with bytecode — an intermediate representation that's neither machine code nor source code. Any platform with a compatible JVM can execute it. The JVM is the abstraction layer — it's platform-specific, but bytecode is not. This is the fundamental design that makes Java portable."

02Primitive vs Reference Types — Memory Deep Dive
Must KnowFrequently Asked
✓
▼
→Primitives — stored directly on the stack (when local variable) or inside the object on heap (when field). Value IS the data.
→Reference types — variable on stack holds a memory address (reference/pointer) that points to the actual object on heap.
// What happens in memory: int a = 5; // stack: [a=5] int b = a; // stack: [a=5, b=5] — copy of value b = 10; // stack: [a=5, b=10] — a unchanged String s1 = "hello"; // stack: [s1 → 0x1A2B] heap: [0x1A2B: "hello"] String s2 = s1; // stack: [s1 → 0x1A2B, s2 → 0x1A2B] same object! // s1 and s2 both point to same heap object
Java is ALWAYS Pass By Value — The Full Explanation
→For primitives — a COPY of the value is passed. Method can't change the original.
→For objects — a COPY of the REFERENCE is passed. Both original and copy point to same object. Method CAN mutate the object's state. But if method reassigns the reference (points to new object), original is unaffected.
void modify(int x) { x = 100; } // primitive — original unchanged void modify(int[] arr) { arr[0] = 100; } // array — original IS changed! void reassign(int[] arr) { arr = new int[]{1,2,3}; } // original unchanged int num = 5; modify(num); // num still 5 int[] arr = {1,2,3}; modify(arr); // arr[0] is now 100 reassign(arr); // arr still {100,2,3} — reassignment inside didn't affect original
How to say it in an interview

"Java is always pass by value. For primitives, a copy of the value is passed. For objects, a copy of the reference is passed — both copies point to the same heap object, so mutations to the object's state are visible, but reassigning the reference inside the method doesn't affect the caller's reference. People confuse this with pass by reference."

04Autoboxing, Unboxing & Integer Cache Trap
Frequently Asked
✓
▼
→Autoboxing — compiler automatically converts primitive to its wrapper class. int → Integer. Happens when assigning primitive to wrapper, adding primitive to collections.
→Unboxing — wrapper class to primitive. Integer → int. Happens in arithmetic expressions. Can throw NullPointerException if wrapper is null!
// Autoboxing List list = new ArrayList<>(); list.add(5); // int 5 is autoboxed to Integer(5) int x = list.get(0); // Integer unboxed to int // NullPointerException from unboxing! Integer nullInt = null; int val = nullInt; // NullPointerException here during unboxing // The Integer Cache Trap — most asked trick question! Integer a = 100; Integer b = 100; System.out.println(a == b); // TRUE — cached range -128 to 127 Integer c = 200; Integer d = 200; System.out.println(c == d); // FALSE — outside cache, different objects! System.out.println(c.equals(d)); // TRUE — always use .equals() for Integers!
Why does the Integer cache exist?

Java caches Integer objects for values -128 to 127 (the most commonly used) to save memory. When you autobox an int in this range, you always get the same cached object. Outside this range, a new Integer object is created each time. This is why == gives different results — it compares object references, not values.

How to say it in an interview

"Autoboxing converts primitives to wrapper objects — the compiler inserts the boxing call automatically. The Integer cache is a well-known trap: Integer values between -128 and 127 are cached, so == returns true. Outside that range, new objects are created, so == returns false. Always use .equals() to compare Integer objects. Also, unboxing a null Integer throws NullPointerException — a common production bug."

String is the single most asked Java topic. Know every corner — pool, immutability, methods, comparison, performance.

01Why String is Immutable & How It's Stored
Must KnowFrequently Asked
✓
▼
Immutability

A String object's value cannot change after creation. The char[] array inside String is marked private final. Any "modification" (concat, replace, substring) creates a brand new String object — the original is untouched.

Why Java made String immutable — 4 reasons
1Security — Strings are used as file paths, class names, DB connection URLs, network addresses. If strings were mutable, a validated URL could be changed after validation. Immutability prevents this attack vector.
2String Pool (memory efficiency) — Multiple variables can safely share the same String object since nobody can change it. If strings were mutable, sharing would be dangerous.
3Thread safety — Immutable objects are inherently thread-safe. No synchronisation needed when multiple threads read the same String.
4hashCode caching — String caches its hashCode after first calculation (lazy init). Since the value never changes, the hashCode never needs recalculation. This makes String an excellent HashMap key.
String s = "hello"; s.toUpperCase(); // creates NEW string "HELLO" — s is still "hello"! System.out.println(s); // "hello" — unchanged String result = s.toUpperCase(); // must capture return value System.out.println(result); // "HELLO" // String is final — cannot be subclassed // String has private final char[] value — cannot be changed after construction
02String Pool — How It Works
Must Know
✓
▼
String Pool

A special region inside the heap (part of Metaspace in Java 8+, was PermGen before) that stores unique string literals. When you write a string literal, JVM first checks if that value exists in the pool. If yes — return the existing reference. If no — create new object in pool and return it.

// Literal — goes to String Pool String a = "hello"; // Pool: ["hello"]. a → pool object String b = "hello"; // Pool already has "hello". b → same object as a String c = "world"; // Pool: ["hello", "world"] System.out.println(a == b); // TRUE — same pool object System.out.println(a == c); // FALSE — different pool objects // new String() — always creates new heap object OUTSIDE pool String d = new String("hello"); // heap object, NOT the pool one System.out.println(a == d); // FALSE — different objects even though same content! System.out.println(a.equals(d)); // TRUE — .equals() compares content // intern() — manually put into pool String e = new String("hello").intern(); // returns pool reference System.out.println(a == e); // TRUE — both point to pool object
How many objects are created?
String s1 = "cat"; // 1 object — in pool (if not already there) String s2 = new String("cat"); // 1 or 2 objects: // "cat" literal may already be in pool (0 new pool objects) // + 1 new heap object = total 1 new object // If "cat" not in pool yet = 2 objects (1 pool + 1 heap) // String concatenation at compile time String s3 = "cat" + "s"; // compiler optimises to "cats" — 1 pool object String s4 = "ca" + new String("ts"); // NOT compile-time, runtime concat — new heap object

Must KnowFrequently Asked

01How HashMap Works Internally
Must KnowFrequently Asked
✓

Internal Structure

HashMap is backed by an array of "buckets" (Node[] table). Default capacity: 16. Each bucket is a linked list (or tree after 8 entries). The index for a key is calculated as: index = hash(key) & (capacity - 1)

Step-by-step: What happens when you call put(key, value)
1Calculate hashCode() of the key
2Apply hash function: hash = hashCode ^ (hashCode >>> 16) — spreads bits to reduce collisions
3Calculate bucket index: index = hash & (capacity-1)
4If bucket is empty — add new Node there
5If bucket has entries — check each via equals(). If key matches, update value. If no match, add to list (collision)
6If a bucket's list grows beyond 8 — convert to balanced tree (Red-Black tree) for O(log n) instead of O(n)
Load Factor & Resizing
→Default load factor: 0.75. When size > capacity × 0.75, resize to 2× capacity
→Resizing is expensive — all entries must be rehashed and redistributed
→If you know size in advance, initialise HashMap with expected capacity to avoid resizing
HashMap vs Hashtable vs ConcurrentHashMap
HashMap Hashtable ConcurrentHashMap
Thread-safe? No Yes (synchronized) Yes (segment locks)
Null key? 1 null key allowed No null key No null key
Performance Fast (no sync) Slow (full lock) Fast (partial locks)
Use when Single-threaded Legacy — avoid Multi-threaded
How to say it in an interview

"HashMap uses an array of buckets. A key's bucket is determined by its hashCode. Collisions are handled via chaining — a linked list per bucket, converted to a Red-Black tree when chain length exceeds 8 for O(log n) lookup. The load factor of 0.75 balances space and lookup performance — when exceeded, the backing array doubles and all entries rehash."

Garbage Collection

GC automatically frees heap memory by removing objects that are no longer reachable from any live thread. An object is "garbage" when no active reference points to it.

equals() & hashCode() Contract

Most important OOP concept for HashMap correctness. VERY frequently asked — know this cold.

01The equals-hashCode Contract
Must KnowFrequently Asked
✓
▼
The Contract (Java specification)

1. If a.equals(b) is true → a.hashCode() MUST equal b.hashCode()
2. If a.hashCode() == b.hashCode() → a.equals(b) MAY be true or false (hash collision is allowed)
3. If you override equals(), you MUST override hashCode() too.

Why does this matter for HashMap?
→HashMap uses hashCode() to find the bucket, then equals() to find the exact key
→If you override equals() but NOT hashCode() — two "equal" objects get different hashCodes → they go to different buckets → map.get() returns null even though the key "exists"
→This is a classic Java bug that's hard to debug
// WRONG — only overrides equals, breaks HashMap class Student { String name; int age; @Override public boolean equals(Object o) { if (this == o) return true; if (!(o instanceof Student)) return false; Student s = (Student) o; return age == s.age && Objects.equals(name, s.name); } // hashCode not overridden — uses Object's identity-based hashCode } Map map = new HashMap<>(); Student s1 = new Student("Alice", 22); map.put(s1, 100); Student s2 = new Student("Alice", 22); System.out.println(s1.equals(s2)); // true System.out.println(map.get(s2)); // null! Different hashCodes → different buckets // CORRECT — override both class Student { String name; int age; @Override public boolean equals(Object o) { if (this == o) return true; if (!(o instanceof Student)) return false; Student s = (Student) o; return age == s.age && Objects.equals(name, s.name); } @Override public int hashCode() { return Objects.hash(name, age); // Java 7+ utility } } // Now map.get(s2) returns 100 correctly
equals() Contract Properties
→Reflexive: x.equals(x) is always true
→Symmetric: if x.equals(y) then y.equals(x)
→Transitive: if x.equals(y) and y.equals(z) then x.equals(z)
→Consistent: same result on repeated calls unless fields change
→Null-safe: x.equals(null) is always false (never throws NPE)
How to say it in an interview

"The contract says: if two objects are equal by equals(), they must have the same hashCode. This is critical for HashMap — it uses hashCode to find the bucket and equals to find the key. Violating this by overriding equals without hashCode causes get() to return null even when the key was put in the map. I always override both together and use Objects.hash() for clean implementation."

Important
✓

→Controls access to a resource pool. acquire() decrements permit count. release() increments it.

→Use case: connection pool, rate limiting, throttling
Semaphore semaphore = new Semaphore(3); // max 3 concurrent threads for (int i = 0; i < 10; i++) { new Thread(() -> { semaphore.acquire(); // blocks if 3 already running try { System.out.println("Accessing resource"); Thread.sleep(1000); } finally { semaphore.release(); // always release } }).start(); } // At most 3 threads run the body simultaneously

→Reusable — can be reset after each cycle (unlike CountDownLatch)
→Use case: parallel computation where all threads must complete phase 1 before any starts phase 2
